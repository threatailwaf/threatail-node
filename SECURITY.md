# Security Policy

## Reporting a vulnerability

Please report security issues privately. Do not open a public issue.

- **GitHub:** use [Private vulnerability reporting](https://github.com/threatailwaf/threatail-node/security/advisories/new) on this repository.
- **Email:** security@threatail.com

Please include the version or commit, a description of the issue, and steps to reproduce it. A proof of concept helps a lot.

### What to expect

| | |
|---|---|
| First response | within 3 business days |
| Assessment and severity | within 7 days |
| Fix for high severity | targeted within 30 days |
| Credit | in the advisory and changelog, unless you prefer otherwise |

We will keep you updated while we work on a fix, and we coordinate disclosure timing with you.

## Scope

This repository covers the Threatail node. Issues in the hosted control plane should be reported the same way and will be routed internally.

Especially relevant here, given what this software does:

- **Detection bypasses** — a request that should be blocked but isn't. These are in scope and we want them.
- Anything allowing the node to be crashed, hung, or made to consume unbounded memory.
- Request smuggling, header or body parsing differentials against common backends.
- SSRF through backend configuration, or escaping the private-address restrictions.
- Any way to make the node forward a request it reported as blocked.

Out of scope: findings that require a malicious configuration file, missing hardening that does not lead to a concrete impact, and reports generated by scanners without a working proof of concept.

## Safe harbor

We will not pursue legal action against researchers who act in good faith, avoid privacy violations and service disruption, and give us reasonable time to fix an issue before disclosing it.

## Advisories

Fixed vulnerabilities are published as GitHub Security Advisories and noted in [CHANGELOG.md](CHANGELOG.md).
